# Version and Tag Policy

## Purpose

This document defines the future CCL version and Git tag policy for releases.
It sets the release identity rules before any tag or release automation exists.

## Current Status

CCL does not yet publish official releases.
CCL does not yet create official Git release tags.
CCL does not yet publish binaries.
CCL is not yet published to crates.io.
This document defines future policy only.

No tag is created by this PR.
No version is bumped by this PR.
No release artifact is generated by this PR.

## Version Format

Future CCL releases should use SemVer-compatible versions in the form:

```text
MAJOR.MINOR.PATCH
```

Recommended policy:

- `0.x.y` while CCL is pre-1.0 and still evolving rapidly;
- `1.0.0` only after stable release workflow, manifest, checksum, and evidence gates exist.

Version meanings:

- MAJOR: incompatible public behavior or release contract changes;
- MINOR: new backwards-compatible features or gates;
- PATCH: bug fixes, docs, compatibility fixes, and small hardening.

Before `1.0.0`, minor versions may still contain breaking changes.
Breaking changes must be documented in release notes and the project ledger.

## Tag Format

Official future release tags should use:

```text
vMAJOR.MINOR.PATCH
```

Examples:

- `v0.1.0`
- `v0.2.0`
- `v0.2.1`
- `v1.0.0`

Reject ambiguous tag formats unless a future policy explicitly adds them:

- `release-0.1`
- `0.1.0`
- `v0.1`
- `latest`
- `stable`

Only `vMAJOR.MINOR.PATCH` tags are release tags.
Other tags are non-release metadata unless explicitly documented.

## Pre-1.0 Compatibility Policy

CCL may evolve quickly before `1.0.0`.
Public CLI behavior should still avoid unnecessary breakage.
Any breaking change must be documented.
Release notes must identify breaking changes.
The ledger must record release warnings.

`0.x` releases are usable early releases, not a long-term stability guarantee.

## Package Version Source

Cargo package versions are source metadata.
Git release tags are release identity.
Release artifacts must record both package version and source commit.

Future rule:

- before an official release tag, Cargo package versions and the Git tag must agree.

Example:

- `Cargo.toml` version `0.1.0`
- Git tag `v0.1.0`

Do not change `Cargo.toml` in this PR.

## Tag Eligibility

A commit may be tagged only if:

- it is on `main`;
- the working tree is clean;
- the local CCL gate passes;
- a release ledger entry exists or is prepared by the release gate;
- release artifact policy is satisfied;
- GitHub CI is checked as metadata only;
- GitHub CI used as evidence: `NO`;
- a human maintainer approves the release decision.

A commit must not be tagged if:

- it is from an unmerged feature branch;
- the working tree is dirty;
- the local CCL gate fails;
- the required ledger entry is missing;
- release artifacts or checksums are inconsistent;
- GitHub CI is the only proof;
- the release evidence is stale or from another commit.

## Evidence Required Before Tagging

Future minimum release evidence should include:

- source commit SHA;
- clean working tree proof;
- local CCL gate manifest;
- validation manifest;
- scope manifest;
- ledger verification manifest;
- admission verdict;
- contract checks;
- `cargo fmt --check`;
- `cargo test`;
- `cargo clippy --all-targets --all-features -- -D warnings`;
- demo script result;
- release ledger entry;
- GitHub CI used as evidence: `NO`.

The intended flow is:

```text
version intent
  -> local CCL release gate
  -> evidence manifests
  -> release ledger entry
  -> human release decision
  -> tag
```

## Dirty Tree Policy

No official tag should come from a dirty working tree.
No official tag should be created while untracked files are present unless they are explicitly ignored and verified.
No official tag should be created from uncommitted local evidence artifacts.

`.ccl/runs/**` artifacts are local evidence outputs and are normally ignored.
They should be referenced in ledger or release manifests when relevant, not committed blindly.

## Branch and Commit Policy

`main` is the only normal release branch for MVP.
Release tags should point to commits reachable from `main`.
Squash-merge commits are acceptable release candidates.
Feature branch heads are not release candidates unless merged.

Future extension:

- `release/*` branches may be introduced later.

## GitHub CI Boundary

GitHub CI is public metadata.
GitHub CI is not release evidence.
A green GitHub check cannot replace local CCL release evidence.
The CI metadata contract is not a replacement for task-specific local admission contracts.

GitHub CI can block a human release decision as a warning signal, but it cannot by itself admit a release.

## Relationship to Release Artifacts

See [`docs/release-artifacts.md`](docs/release-artifacts.md).

Version and tag policy must exist before official release artifacts.
Release artifacts must reference source commit and version.
Checksums must be generated from final artifact bytes.
The release manifest must bind version, tag, artifacts, checksums, and evidence.

Do not generate artifacts in this PR.

## Future Release Flow

Future release sequence:

1. Decide intended version.
2. Confirm version policy.
3. Run local CCL release gate.
4. Build release artifacts.
5. Generate checksums.
6. Generate release manifest.
7. Update release ledger entry.
8. Human reviews evidence.
9. Create tag `vMAJOR.MINOR.PATCH`.
10. Draft GitHub Release.
11. Publish only after evidence review.

This is future design.
Current PR does not implement this flow.

## Non-goals for Current MVP

- tag creation;
- version bumping;
- release artifact generation;
- release automation;
- GitHub Release publishing;
- checksums;
- manifest generation;
- signature or signing workflows;
- crates.io publishing;
- runtime changes;
- branch protection.

## Common Mistakes

- treating GitHub CI as release evidence;
- tagging a branch before the local CCL gate passes;
- creating a tag before release artifacts and checksums are finalized;
- using a version tag that does not match Cargo package versions;
- reusing a tag after artifacts changed;
- tagging from a dirty working tree;
- treating `.ccl/runs/**` as release artifacts.
